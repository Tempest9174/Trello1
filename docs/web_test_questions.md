# 第1章 インターネットとネットワークの仕組み（12問）

## Q1. インターネットとは

「インターネット」と「Webブラウザで見るホームページ」は同じものですか？違うとすれば、どう違いますか？

### 回答

違います。インターネットとは世界中に広がっているクモの巣状のネットワークのようなもので世界中のコンピュータ同士をつなぐネットワークの仕組みです。ホームページはインターネット上に公開されているサイトのことです。HTTP/HTTPSというプロトコルを使って閲覧するものです。

## Q2. IPアドレスとポート番号  このプロジェクトではフロントエンドを `localhost:5173`、バックエンドを `localhost:8080` で起動しています。  - `localhost`とは何を指しますか？ - `:5173` や `:8080` のようなポート番号はなぜ必要ですか？

### 回答

ローカルホストとは主に開発用に自分のPCをサーバーとして使う自分のコンピュータのことを表します。ローカルホストは特別なIPアドレスがあり127.0.0.1というIPアドレスを持ちます。ポート番号がなぜ必要かはパソコンが1台のコンピュータは複数のアプリが同時に通信を待ち受けているため、IPアドレスだけではどのアプリへの通信か特定できないそれを認識区別するためポート番号を指定しています。

## Q3. DNS  ブラウザのアドレスバーに  [https://www.example.com](https://www.example.com)  と入力してからページが表示されるまでに、DNSはどのような役割を果たしますか？

### 回答

DNSはもともとホームページのアドレスが数字で表される無意味な羅列なため分かり辛いです。それを人間にもわかりやすくするように先のwww.example.comと変換する役割を果たしています。通信するときにはIPアドレスという数字のアドレスですが人間が指定のアドレスページにアクセスする際にはexample.comと入力することでページが表示されます。

## Q4. HTTPとは

HTTPとは何の略で、どのような役割を持つものですか？

「プロトコル」という言葉の意味も含めて説明してください。

### 回答

HTTPはハイパーテキスト・トランスファープロトコルの略です。役割はWebサーバーとクライアント（ブラウザ）同士がデータをやり取りするためのルールです。プロトコルとは通信における約束事・手順のことです。
「プロトコル（protocol）」とは、通信における約束事・手順 になります。人間どうしの会話でも「日本語で話す」「最後まで聞いてから返事する」といった共通のルールがあるように、コンピュータどうしが正しく情報をやり取りするには、お互いに従う決まりが必要です。この決まりを取り決めたものがプロトコルです。

## Q5. HTTPとHTTPSの違い

HTTPとHTTPSの違いは何ですか？

なぜ現在のWebサイトはほぼすべてHTTPSを使用していますか？

### 回答

HTTPとHTTPSの違いは、通信内容が暗号化されているかどうかです。HTTPはWebブラウザとサーバーがデータをやり取りするための約束事だと説明しました。HTTPSは、その通信に暗号化という安全のしくみを加えたものです。末尾のSは Secure(セキュア＝安全) を意味します。
HTTPは通信内容が暗号化されずにそのまま送信されます。HTTPSは通信内容が暗号化されて安全に送信されます。

## Q6. リクエストとレスポンス

Webにおける「リクエスト」と「レスポンス」の関係を、日常生活の例えを使って説明してください。

### 回答

リクエストとレスポンスはよく料理店で注文するときのお客さんと店員の関係のようなものです。リクエストはメニューで料理を店に注文することです。レスポンスは料理をお客提供することです。

## Q7. HTTPメソッド  このアプリのAPIでは以下を使っています。  - GET - POST - PUT - DELETE - PATCH  これらの「HTTPメソッド」とは何ですか？  日常生活の例えで、それぞれの違いを説明してください。

### 回答

HTTPメソッドはHTTPリクエストの種類を表すものです。住所録の喩えで説明するとGETは登録されているAさんの電話番号を確認するイメージのメソッドです。POSTは住所録に新しく乗ってない人を書き加えるためのメソッドです。PUTは住所をまるごと更新するためのメソッドです。例えばBさんの住所を更新するときにはBさんの名前・住所・電話まるごと更新するのがPUTです。DELETEは住所録を削除するためのメソッドです。例えばCさんの住所録を削除するときにはCさんの住所などを丸ごと捨てるイメージです。PATCHは住所録を一部だけ更新するためのメソッドです。例えばDさんを更新するときにはDさんの電話番号だけ新しく更新し他の情報はそのままにするメソッドです。

## Q8. HTTPステータスコード  このアプリのAPIでは以下を使っています。  - 200 - 201 - 400 - 404 - 500  これらの「HTTPステータスコード」とは何ですか？  日常生活の例えで、それぞれの違いを説明してください。

### 回答

ステータスコードとは、サーバーが返事をするときに付ける「今の状況を表す3桁の番号」です。200（OK） は「成功しました」という意味です。201（Created） は「新しく作成できました」という意味です。何かを新規に登録・作成したときに返ってきます。400（Bad Request） は「あなたの頼み方がおかしいです」という意味です。404（Not Found） は「そんなものは見つかりません」という意味です。頼まれたものが存在しない状態です。500（Internal Server Error） は「サーバー側で問題が起きました」という意味です。頼み方は正しかったのに、サーバー内部のトラブルで処理できなかった状態です。

## Q9. JSONとは  フロントエンドとバックエンド間でデータをやり取りする際に「JSON」という形式が使われています。  - JSONとは何ですか？ - なぜ広く使われていますか？

### 回答

JSONとはJavaScript Object Notationの略で、コンピュータ同士がデータをやり取りするためのフォーマットでデータを表現するテキスト形式で、キーと値のペアで書きます。なぜ広く使われているのは理由はいくつかあります。人間にとって読みやすく書きやすいからです。又コンピュータにとってもデータを解析しやすいからです。

## Q10. APIとは

### 回答

APIとはApplication Programming Interfaceの略で、APIは「プログラムどうしをつなぐ決められた窓口」であり、データベースに直接アクセスしないのは、安全を守り、データを壊させず、変更や管理をしやすくするためなのです。

## Q11. CORSとは  フロントエンド（5173）からバックエンド（8080）にアクセスする際、  「CORS」という仕組みが関係します。  CORSとは何のためにありますか？

### 回答

CORSとは Cross-Origin Resource Sharing（クロスオリジン・リソース・シェアリング） の略です。日本語にすると「異なるオリジン（出どころ）の間で、リソースのやり取りを許可する仕組み」という意味になります。
フロントエンド（5173）とバックエンド（8080）はポート番号が違うため別オリジンとみなされ、何もしなければブラウザがこの通信をブロックしてしまいます。それを許可する仕組みがCORSです。

## Q12. クライアントとサーバー

「クライアント」と「サーバー」とは何ですか？

このアプリの構成の中で、それぞれどれがクライアントでどれがサーバーですか？

- ブラウザ
- バックエンド
- データベース

### 回答

サーバーとは要求を受け、答える側です。内容を処理し結果を返す役割を持ちます。クライアントとはサービスをお願いする側です。サーバーとの関係で言えばその要求を出す側です。

- ブラウザはクライアントです。ユーザーが操作しデータを下さいという要求を出す側です。
- バックエンドはサーバーです。ユーザーが操作しデータを下さいという要求を受け、内容を処理し結果を返す役割を持ちます。
- データベースはサーバーです。バックエンドからのデータを取り出して、保存してというお願いを受け、データを提供・保存して返す役割を持ちます。

# 第2章 Webアプリケーションの構造（8問）

## Q13. フロントエンドとバックエンド  「フロントエンド」と「バックエンド」の違いを説明してください。  このアプリではそれぞれ何を担当していますか？

### 回答

フロントエンドとバックエンドは、Webアプリを「目に見える画面側」と「裏側で動く処理部分」に分けたときの呼び方となります。
フロントエンド は、ユーザーが直接見て触れる画面側を担当します。ボタンやデザインの表示、クリックやドラッグへの反応などを受け持ち、主にブラウザの中で動きます。
バックエンド は、ユーザーからは見えない裏側を担当します。データの保存や取り出し、ログイン認証、各種処理の判断などを受け持ち、サーバー側で動きます。
今回のTrelloアプリでの担当を整理すると、こうなります。
フロントエンドは、ボードやカードを画面に表示すること、カードをドラッグして動かす操作、クリックで詳細を開く反応など、見て・触って・操作する部分を担当します。
バックエンドは、作られたカードの内容をデータベースに保存すること、次に開いたときにカードを取り出して返すこと、ユーザーの権限の判断など、「データの管理や処理」を担当します。

## Q14. 3層構造  このアプリは  - フロントエンド - バックエンド - データベース  の3層構造です。  なぜこのように役割を分けるのですか？  1つのプログラムですべてをやらない理由を説明してください。

### 回答

役割を分ける理由

問題の特定がしやすい:画面がおかしければフロントエンド、データがおかしければデータベースと、修正箇所が明確になる
変更の影響を限定できる:デザインだけ変えたいときはフロントエンドだけ触ればよく、他の層に影響が及ばない
複数人で分業できる:層ごとに担当を分けて並行して開発できる

## Q15. データの流れ  ユーザーが「新しいタスクを作成」ボタンを押してから、  - データベースに保存される - 画面に反映される  までの流れを順を追って説明してください。

### 回答

ユーザーが「新しいタスクを作成」ボタンを押すとフロントエンドがバックエンドにPOSTリクエストを送ります。

データベースがバックエンドに保存したタスクデータを返す。バックエンドがフロントエンドに保存したタスクデータを返す。フロントエンドが画面にタスクデータを表示します。

## Q16. エンドポイントとURL設計  このアプリには以下のAPIがあります。  - /api/tasks - /api/tasks/{id}  このようなURL設計の考え方を説明してください。  {id} の部分は何ですか？

### 回答

エンドポイントとは、APIにアクセスするための入り口を示すURLです。クライアントはこのURLで「どのリソース（操作対象）を扱いたいか」を指定します。/api/tasksはタスク全体を表し、一覧取得や新規作成の対象になります。/api/tasks/{id}は/api/tasks/5のように特定の1件を表し、その取得・更新・削除の対象になります。これら「URLは名詞、操作はメソッド、構造は階層と一貫性で表す」という設計原則を体系化したものを REST（REpresentational State Transfer） と呼び、Web APIの事実上の標準として広く採用されています。

## Q17. 同期処理と非同期処理  「同期処理」と「非同期処理」の違いを、日常生活の例えで説明してください。  なぜWeb通信では非同期処理が必要ですか？

### 回答

同期処理と非同期処理は、処理の完了を待つか待たないかの違いです。同期処理は処理が完了するまで待ち、非同期処理は処理が完了するのを待たずに次の処理を行います。日常生活の例えでは、料理店で注文をするときには、料理が完成するまで待ちます。これは同期処理です。一方、料理店で注文をするときには、料理が完成するのを待たずに次の料理を注文することもできます。これは非同期処理です。

Web通信でなぜ非同期処理が必要なのかは、通信が完了するのを待たずに次の処理を行いたいからです。例えば、ユーザーがボタンをクリックしてから、画面に反映されるまでの間に、別のボタンをクリックしても、別のボタンが反応することがあります。これは、通信が完了するのを待たずに次の処理を行ったためです。また、もし同期処理にすると、その待ち時間ずっと画面が固まってしまうからです。データが返ってくるまで他の操作を一切受け付けないとなると、ユーザーはボタンも押せず、スクロールもできず、アプリが「フリーズ？した」ように感じてしまうからです。

## Q18. SPA（シングルページアプリケーション）  このアプリはReactで作られたSPAです。  従来のWebサイト（ページ遷移するもの）と何が違いますか？

### 回答

SPAは Single Page Application（シングル・ページ・アプリケーション） の略で、1枚のページの中身を部分的に書き換えながら動くWebアプリのことです。
従来のWebサイト（ページ遷移型）
従来のサイトは、リンクをクリックするたびに、サーバーから新しいページ全体を丸ごと受け取って表示し直す仕組みです。これをページ遷移といいます。
SPA（シングルページ型）
SPAは、最初に1枚のページを読み込んだら、あとは必要な部分だけをその場で書き換える仕組みです。ページ全体を取り直さず、変わるところだけを差し替えます。
SPAの仕組みとしてサーバーとはデータだけをやり取りすることです。画面が変わるたびにページ全体をもらうのではなく、必要なデータ（JSON）だけをAPIで取得し、それを使って画面の一部を更新します。やり取りが軽くなり、動きが速くなります。

## Q19. 入力チェック（バリデーション）の多重防御  入力チェックはフロントエンドとバックエンドの両方で行うべきと言われています。  なぜ片方だけでは不十分なのですか？

### 回答

フロントエンドだけでは不十分 — フロントのチェックはブラウザ内で動くため、開発者ツールを使ったり、ブラウザを通さず直接APIに送ったりすれば簡単に回避できるからです。鍵のない裏口があるのに玄関だけ施錠している状態で、不正なデータがそのままサーバーに届いてしまいます。
バックエンドだけでは不十分 — 入力ミスのたびにサーバーへ送って返事を待つ必要があり、反応が遅く使い勝手が悪いからです。
役割分担 — フロントは「使いやすさ」担当（その場で即座にミスを指摘する）、バックは「安全性」担当（回避できない最後の砦になる）。

## Q20. エラーハンドリング  API呼び出しが失敗した場合（ネットワーク障害やサーバーエラーなど）、  アプリはどのように対処すべきですか？

### 回答

まず失敗のパターンを分けて考えます。1. ネットワーク障害:通信が途絶える2. サーバーエラー:サーバーが異常な状態3. 404エラー:存在しないデータを要求4. 400エラー:不正なリクエスト5. 500エラー:サーバー内部エラーなどです。
対処法としてはユーザーにわかりやすく状況を伝えることです。画面が無反応なままだとユーザーは何が起きているかわからないからです。なにが起きたのか、どうすればいいか伝えます。その際技術的な詳細はそのまま見せないことがポイントです。また、データを壊さない・整合性を保つことも重要です。たとえばTrelloでカードを動かして保存に失敗したら、画面だけ動いたままにせず、元の位置に戻すなどして、画面とデータベースの食い違いを防ぎます。
開発者としては、ログに詳しく残します。 いつ起きたか（日時）、どのAPIで起きたか、どんなエラーだったか、といった情報を記録しておけば、後から「なぜ失敗したのか」を追跡できます。

## Q21. データベースとは

データベースとは何ですか？

データベースは、大量のデータを、速く・安全に・複数人で・正しく扱うために作られた仕組みです。「探す・絞る・並び替え」といった操作を効率よく行えます。少人数で少量のデータを扱うだけならExcelやCSVでも足りますが、Trelloアプリのように多くの人が同時に使い、データの正しさが求められるアプリでは、データベースが欠かせないです。

tasksテーブルを表に例えると
カラム（列）が見出しとして並び、その下に1件ごとのタスクが行として積み重なります。タスクテーブルを例にすると

## Q25. データの制約

tasksテーブルでは

- `NOT NULL`
- `DEFAULT 'medium'`

が設定されています。

これらは何のためにありますか？

### 回答

NOT NULL — 「空っぽを禁止する」

タスクを別カラムへドラッグ移動する操作で、なぜ重要ですか？

### 回答

トランザクション(transaction)とは、複数の処理を「ひとまとまり」として扱い、全部成功するか、全部なかったことにするか、どちらかにする仕組みです。「全部できる」か「全部やらない」のどちらかで、途中半端な状態を絶対に残さないのがポイントです。このことを専門的には原子性と呼ぶ。

## Q28. SQLとORM

- SQLとはどのようなものですか？
- ORMの役割とは何ですか？

### 回答

SQL(エスキューエル)とは、データベースに指示を出すための専用の言語です。「このデータを取り出して」「これを保存して」「これを消して」といった命令を、データベースが理解できる形で書くための言葉です
ORMとは、SQLでなくプログラミング言語のコードを書くだけで、その裏でSQLを自動的に生成・実行してくれる仕組みです。Object-Relational Mapping(オブジェクト関係マッピング) の略です。

# 第4章 開発プロセスとチーム開発（4問）

## Q29. バージョン管理（Git）

Gitなどのバージョン管理システムとは何ですか？

なぜ必要ですか？

使わないとどんな問題が起きますか？

### 回答

Gitなどのバージョン管理システムとは、ファイルの変更履歴を記録し、いつでも過去の状態に戻せるようにする仕組みです。プログラムを書いていく過程を1段階ずつ「セーブ」していくイメージで、ゲームのセーブポイントのように、何かトラブルが起きた時や好きなときに以前の状態を呼び出せます。
なぜ必要かというと、まず変更を記録して、いつでも戻せるからです。新しく書き換えて失敗しても、前のセーブ地点に戻せるので、安心してプログラムが試せます。もし使わないと失敗したとき手作業で直すしかなくなります。要するに「戻せない、最新が不明になる、など問題が起きます。

## Q30. ブランチとPull Request

- ブランチとは何ですか？
- Pull Request の目的は何ですか？
- なぜ全員が main に直接変更しないのですか？

### 回答

ブランチ → mainから枝分かれした独立の作業スペースmainに影響を与えず作業できる。
Pull Request → 「mainに取り込んで」という提案。いきなり合流させず、まず他のメンバーや上司に内容を見てもらい、おかしな点や危険な書き方がないかをコードレビューしてもらう。問題がなければ承認され、mainに統合される。
直接変更しない理由 → mainを常に動く安全な状態に保つため。全員が直接書き込むと、未完成や不具合のあるコードがそのまま本体に混ざったり、複数人の変更が衝突して上書き事故が起きたりするため

## Q31. コミットメッセージの重要性

例：

`feat: ログイン機能を追加`

良いコミットメッセージを書くことがなぜ重要ですか？

### 回答

コミットメッセージとは、変更をコミットするときに添える「何をしたか」の説明です。
セーブしたときに付けるメモ書きにあたる。
良いコミットメッセージが重要なのは、まず何を変更したかが一目でわかるからです。feat: ログイン機能を追加のように書いてあれば、中身を開かなくても、その変更で何をしたかが一目で伝わります。

## Q32. なぜ技術的な仕組みを理解すべきか

AI時代においてコードはAIが書けるようになりました。

それでも

- ネットワーク
- データベース
- Webアプリケーションの仕組み

を理解する必要がある理由を、あなたの考えで述べてください。

### 回答

AIはコードを書けますが、そのコードが実際の環境で起こす問題まですべて見通せるわけではありません。トラブルは、コードの文面だけ眺めても原因が分かりません。ネットワークのどこで止まったのか、データベースで何が起きたのか、層のどこに問題があるのかを切り分けて初めて原因にたどり着けます。そのような仕組みを理解している人がAIを使うと、生産性は飛躍的に上がります。AIに的確な指示を出せ、出力の良し悪しを即座に判断でき、問題が起きても自分で原因を切り分けられます逆に理解のない人がAIを使うと、出力を鵜呑みにするしかなく、何かあれば手も足も出ません。
