## 1. 大きな概念から理解しよう

### フロントエンド（Frontend）

> **一言で言うと：** ユーザーが実際に見て触る「画面」の部分

ブラウザ（ChromeやSafariなど）上に表示されるもの全て。  
ボタン・テキスト・カード・色などの見た目と、クリックなどの操作を担当する。

```
ユーザー → ブラウザ → フロントエンド（React）
             ここがフロントエンド ↑
```

### バックエンド（Backend）

> **一言で言うと：** ユーザーには見えない「サーバー側」の処理

データの保存・取得・計算などのロジックを担当する。  
フロントエンドから「このデータを保存して」と頼まれたら処理する係。

```
フロントエンド → バックエンド（Spring Boot）→ データベース
                  ここがバックエンド ↑
```

### サーバー（Server）

> **一言で言うと：** 常に起動していてリクエストを待ち受けているコンピュータ（またはプログラム）

「サーバー」と聞くと大きな機械を想像するかもしれないが、  
開発中は自分のパソコン上でサーバープログラムを動かすことが多い（= ローカルサーバー）。

### クライアント（Client）

> **一言で言うと：** サーバーにリクエストを送る側

ブラウザ（フロントエンド）がクライアントになることが多い。  
「お客さん」と「お店」の関係で言うと、クライアントが「お客さん」、サーバーが「お店」。

## 2. フロントエンドの用語

### JavaScript

> **一言で言うと：** ブラウザで動くプログラミング言語

ウェブページに動きをつけるための言語。  
ボタンを押したときに何かが起きる、入力した文字が表示されるなどの動作を作れる。  
ブラウザが標準で理解できる唯一のプログラミング言語。

### TypeScript（タイプスクリプト）

> **一言で言うと：** JavaScriptに「型チェック」を追加したもの

JavaScriptの上位版。  
「この変数は数字だよ」「この関数は文字列を返すよ」と明示できる。

```
// JavaScript（型がないので間違えやすい）
let name = "田中";
name = 123;  // 数字を入れてしまっても気づけない

// TypeScript（型があるのでミスを検知できる）
let name: string = "田中";
name = 123;  // エラー！文字列型に数字は入れられません、と教えてくれる
```

### React（リアクト）

> **一言で言うと：** 画面のパーツ（部品）を作るためのJavaScriptライブラリ

Metaが作ったUI構築ツール。  
画面を「ボードカード」「リスト列」「カード」のような**部品（コンポーネント）**に分けて作れる。

```
// Reactのコンポーネントの例
function CardItem({ title }) {
  return <div>{title}</div>;  // HTMLのようなコードが書ける（JSXという書き方）
}
```

### コンポーネント（Component）

> **一言で言うと：** 画面の「部品」

Trelloで言えば「カード1枚」「リスト1列」「ボタン1つ」などが1つのコンポーネント。  
同じ部品を複数の場所で使い回せる（例：カードコンポーネントを100個表示など）。

### Vite（ヴィート）

> **一言で言うと：** Reactアプリを素早く起動・ビルドするツール

`npm run dev` でローカルサーバーを起動したり、  
`npm run build` で本番用ファイルを生成したりする。  
「開発用の足場を組んでくれる便利な台車」のようなもの。

### Tailwind CSS（テイルウィンドCSS）

> **一言で言うと：** HTMLのクラス名を書くだけでスタイルが決まるCSSツール

通常CSSはこう書く：
```css
/* 通常のCSS */
.button {
  background-color: blue;
  padding: 8px 16px;
  border-radius: 4px;
}
```

Tailwindを使うとHTMLに直接書ける：
```html
<!-- Tailwind CSS -->
<button class="bg-blue-500 px-4 py-2 rounded">
  クリック
</button>
```

### dnd-kit（ディーエヌディーキット）

> **一言で言うと：** ドラッグ＆ドロップを実装するライブラリ

「ライブラリ」とは他人が作った便利な機能の集まり。  
自分でドラッグ処理を一から作るのは非常に難しいため、このライブラリを使う。  
Trelloのカードを掴んで移動する動作を実現する。

### Zustand（ズスタンド）

> **一言で言うと：** Reactアプリ全体でデータを共有・管理するライブラリ

Reactは部品（コンポーネント）ごとにデータを持っているが、  
「どのカードが選択されているか」など**アプリ全体で共有したいデータ**はここで管理する。

```
例：カードモーダルを開く
  CardItem（部品A）でクリックが起きる
    ↓
  Zustandのストアに「モーダルを開く」と記録
    ↓
  CardModal（部品B）がストアを見て自分を表示する
```

### Axios（アクシオス）

> **一言で言うと：** JavaScriptからサーバーへHTTPリクエストを送るライブラリ

フロントエンドからバックエンドAPIに「データを取得して」「保存して」と頼むための道具。

```javascript
// Axiosを使ってボード一覧を取得する例
const response = await axios.get("http://localhost:8080/api/boards");
console.log(response.data);  // [{ id: 1, title: "仕事用ボード" }, ...]
```

### npm（エヌピーエム）

> **一言で言うと：** JavaScriptのライブラリを管理するツール

Node Package Manager（ノード・パッケージ・マネージャー）の略。  
`npm install axios` と打つだけでAxiosをダウンロードして使えるようにしてくれる。  
Pythonの `pip`、Javaの `Maven` に相当するもの。

### Node.js（ノードジェイエス）

> **一言で言うと：** JavaScriptをブラウザ以外（パソコン上）で動かせる環境

通常JavaScriptはブラウザ上でしか動かせないが、  
Node.jsをインストールするとパソコン上で直接実行できる。  
ViteやnpmはNode.jsの上で動いている。

## 3. バックエンドの用語

### Java（ジャバ）

> **一言で言うと：** 企業システムでよく使われるプログラミング言語

「1度書けばどこでも動く」が特徴。  
型が厳格でバグが見つかりやすく、大規模なシステム開発に向いている。  
このプロジェクトではバックエンド（サーバー側）の処理を書くために使う。

### Spring Boot（スプリングブート）

> **一言で言うと：** JavaでWebサーバー・APIを簡単に作れるフレームワーク

「フレームワーク」とは「アプリを作るための骨格（ひな形）」のこと。  
自分でゼロからHTTPサーバーを作るのは大変だが、  
Spring Bootを使えば最低限のコードでAPIが作れる。

```java
// Spring Bootのコントローラーの例
@RestController
@RequestMapping("/api/boards")
public class BoardController {

    @GetMapping          // GET /api/boards にアクセスされたときの処理
    public List<Board> getAllBoards() {
        return boardService.findAll();
    }
}
```

### フレームワーク（Framework）

> **一言で言うと：** アプリを作るための「型（かた）」や「骨格」

白紙からアプリを作るのではなく、よく使われる仕組みがあらかじめ組み込まれている。  
料理で言えば「ルーから作るカレー」が自作、「カレーの素」がフレームワークを使う感覚。

### ライブラリ（Library）

> **一言で言うと：** 特定の機能を提供する「部品集」

フレームワークより小さい単位。  
ドラッグ＆ドロップの機能だけほしい → dnd-kitライブラリを使う、という感じ。  
フレームワークが「家の骨組み」なら、ライブラリは「ドアや窓」。

### Maven（メイヴン）

> **一言で言うと：** Javaのライブラリをダウンロードしてビルドするツール

JavaScriptの `npm` に相当するもの。  
`pom.xml` というファイルに「このライブラリを使いたい」と書くと、  
自動でインターネットからダウンロードしてくれる。

### Spring Data JPA（スプリングデータジェイピーエー）

> **一言で言うと：** JavaのクラスとDBのテーブルを自動で対応させるORM

「ORM」（Object Relational Mapping）とは、  
プログラムのオブジェクト（クラス）とDBのテーブルを紐付けること。

```
通常（SQL直書き）:
  "SELECT * FROM boards WHERE id = 1"  ← SQLを書く必要がある

Spring Data JPAを使うと:
  boardRepository.findById(1)  ← Javaのメソッド呼び出しでOK
```

### Lombok（ロンボク）

> **一言で言うと：** Javaの面倒な定型コードを自動生成するライブラリ

Javaでは変数の値を取得・設定するための「getter/setter」というメソッドを全変数分書く必要があるが、  
`@Data` というアノテーション1つでそれらを全て自動生成してくれる。

```java
// Lombokなし（全部自分で書く）
public class Board {
    private Long id;
    private String title;

    public Long getId() { return id; }
    public void setId(Long id) { this.id = id; }
    public String getTitle() { return title; }
    public void setTitle(String title) { this.title = title; }
}

// Lombokあり（@Dataだけでゲッター・セッターが自動生成）
@Data
public class Board {
    private Long id;
    private String title;
}
```

### アノテーション（Annotation）

> **一言で言うと：** コードに付ける「タグ」や「メモ」

`@` で始まる記法。「このクラスはコントローラーですよ」「このメソッドはGETリクエストを処理します」  
などの意味を付与する。Spring Bootでは大量のアノテーションを使う。

```java
@RestController    // ← これがアノテーション（このクラスはREST APIのコントローラーです、という意味）
public class BoardController {
    @GetMapping("/api/boards")    // ← これもアノテーション
    public List<Board> getBoards() { ... }
}
```

## 4. データベースの用語

### データベース（Database / DB）

> **一言で言うと：** データを永続的に保存しておく場所

プログラムが終了してもデータが消えないようにするための保管庫。  
Excelのような表形式でデータを管理する（リレーショナルデータベース）。

### MySQL（マイエスキューエル）

> **一言で言うと：** 広く使われているリレーショナルデータベース管理システム

データをテーブル（表）形式で保存する。  
このプロジェクトでは「boards」「cards」などのテーブルを作る。

### テーブル（Table）

> **一言で言うと：** データを格納するExcelのような表

```
boardsテーブルの例:
┌────┬─────────────┬──────────┬─────────────────────┐
│ id │    title    │background│      created_at     │
├────┼─────────────┼──────────┼─────────────────────┤
│  1 │  仕事用     │ #0079BF  │ 2026-04-25 10:00:00 │
│  2 │  個人用     │ #70B500  │ 2026-04-25 11:00:00 │
└────┴─────────────┴──────────┴─────────────────────┘
```

### カラム（Column）/ フィールド（Field）

> **一言で言うと：** テーブルの「列」（縦方向）

上の表で言うと `id`, `title`, `background`, `created_at` がカラム。  
各カラムには「文字列型」「数値型」「日付型」などの**データ型**が決まっている。

### レコード（Record）/ 行（Row）

> **一言で言うと：** テーブルの「1行分のデータ」

上の表で言うと `1 | 仕事用 | #0079BF | ...` が1レコード。  
ボード1つ、カード1枚がそれぞれ1レコードに対応する。

### 主キー（Primary Key / PK）

> **一言で言うと：** レコードを一意に識別するID

`id` カラムがこれにあたる。  
同じIDを持つレコードは存在できない（重複不可）。  
「社員番号」のようなもので、他のテーブルからそのレコードを参照するときに使う。

### 外部キー（Foreign Key / FK）

> **一言で言うと：** 別テーブルのレコードへの参照

`board_lists` テーブルの `board_id` がこれにあたる。  
「このリストはどのボードに属しているか」を示す。

```
boardsテーブル             board_listsテーブル
┌────┬────────┐           ┌────┬──────────┬─────────────┐
│ id │ title  │           │ id │ board_id │    title    │
├────┼────────┤           ├────┼──────────┼─────────────┤
│  1 │ 仕事用 │←──────────│  1 │    1     │ やること    │
│  2 │ 個人用 │           │  2 │    1     │ 進行中      │
└────┴────────┘           │  3 │    2     │ TODO        │
                          └────┴──────────┴─────────────┘
                                   ↑外部キー（board_idは1か2）
```

### CASCADE（カスケード）

> **一言で言うと：** 親を削除したら子も自動で削除する設定

`ON DELETE CASCADE` と書いておくと、  
ボードを削除したら → そのボードのリストも自動削除 → そのリストのカードも自動削除、と連鎖する。

### SQL（エスキューエル）

> **一言で言うと：** データベースを操作するための専用言語

```sql
-- データを取得する
SELECT * FROM boards;

-- データを追加する
INSERT INTO boards (title) VALUES ('新しいボード');

-- データを更新する
UPDATE boards SET title = '変更後タイトル' WHERE id = 1;

-- データを削除する
DELETE FROM boards WHERE id = 1;
```

### ENUM型（イーナム型）

> **一言で言うと：** 決まった選択肢の中からしか値を入れられない型

カードの優先度（priority）に使っている。

```sql
priority ENUM('HIGH', 'MEDIUM', 'LOW')
-- HIGH、MEDIUM、LOW の3つ以外は入れられない
```

## 5. APIと通信の用語

### API（エーピーアイ）

> **一言で言うと：** ソフトウェア同士が会話するための「窓口」

Application Programming Interface の略。  
フロントエンドがバックエンドにデータを頼む際のルールや窓口のこと。

「レストランで料理を注文するときのメニュー」のようなもの。  
メニュー（API）に書かれた方法でオーダー（リクエスト）すれば、料理（レスポンス）が返ってくる。

### REST API（レストAPI）

> **一言で言うと：** URLとHTTPメソッドでAPIを設計するスタイル

`GET /api/boards` → ボード一覧を取得  
`POST /api/boards` → ボードを新規作成  
このように「URLで対象」「HTTPメソッドで操作」を表す設計ルール。

### HTTPメソッド

> **一言で言うと：** リクエストの「目的・操作の種類」

| メソッド | 意味 | 例 |
|---------|------|-----|
| **GET** | データを**取得**する | ボード一覧を見る |
| **POST** | データを**新規作成**する | 新しいボードを作る |
| **PUT** | データを**まるごと更新**する | ボードのタイトルを変更する |
| **PATCH** | データを**一部だけ更新**する | カードの並び順だけ変える |
| **DELETE** | データを**削除**する | ボードを削除する |

### HTTPステータスコード

> **一言で言うと：** リクエストの結果を数字で表したもの

| コード | 意味 |
|--------|------|
| **200** | 成功 |
| **201** | 作成成功（POSTが成功したとき） |
| **400** | リクエストが不正（入力値が間違い） |
| **404** | 見つからない（指定したIDが存在しない） |
| **500** | サーバー内部エラー |

### JSON（ジェイソン）

> **一言で言うと：** データをやり取りするときの書き方（フォーマット）

JavaScript Object Notation の略。  
フロントエンドとバックエンド間でデータを送受信するときに使う形式。

```json
{
  "id": 1,
  "title": "仕事用ボード",
  "background": "#0079BF",
  "createdAt": "2026-04-25T10:00:00"
}
```

### CORS（コルス）

> **一言で言うと：** 異なるオリジン（URLのドメイン＋ポートの組み合わせ）間の通信を許可する仕組み

ブラウザはセキュリティのため、デフォルトで「自分と違うオリジン（アドレス）へのリクエスト」をブロックする。

```
フロントエンド: http://localhost:5173
バックエンド:   http://localhost:8080  ← ポートが違う → 別オリジン → ブロックされる！

→ Spring Boot側でCORS設定をして「localhost:5173からのリクエストは許可します」と明示する
```

### ポート番号（Port）

> **一言で言うと：** 同じコンピュータ上の「どのプログラムへの通信か」を区別する番号

`localhost:8080` の `:8080` の部分。  
同じPCでフロントとバックを両方起動するとき、ポートで区別する。

```
http://localhost:5173 → Vite（Reactの開発サーバー）
http://localhost:8080 → Spring Boot（JavaのAPIサーバー）
http://localhost:3306 → MySQL（データベース）
```

### localhost（ローカルホスト）

> **一言で言うと：** 自分のコンピュータ自身を指すアドレス**

`127.0.0.1` と同じ意味。「自分のPC内で完結した通信」に使う。  
開発中はインターネットに公開せず、自分のPC内だけで動かす。

### エンドポイント（Endpoint）

> **一言で言うと：** APIの「窓口のアドレス」

`GET /api/boards` や `POST /api/cards` のように、  
「どのURLに」「どのHTTPメソッドで」アクセスするかを指定したもの。

### リクエスト（Request）

> **一言で言うと：** クライアントからサーバーへの「お願い」

「ボード一覧をください」「このカードを保存してください」などの要求。

### レスポンス（Response）

> **一言で言うと：** サーバーからクライアントへの「返答」

リクエストに対してサーバーが返すデータ（JSONなど）や結果コード。

### 楽観的更新（Optimistic Update）

> **一言で言うと：** サーバーの返答を待たずに先に画面を更新する手法

ドラッグ＆ドロップでカードを移動するとき、  
サーバーへのリクエスト完了を待つと画面の反応が遅れてしまう。  
「たぶん成功するだろう」と楽観的に考えて先に画面を動かし、  
もし失敗したら元に戻す（ロールバック）。

## 6. 開発ツールの用語

### IDE（アイディーイー）

> **一言で言うと：** コードを書くための専用エディタ

Integrated Development Environment（統合開発環境）の略。  
VS Code、IntelliJ IDEAなどが有名。  
コードの色付け・補完・エラー表示などの便利機能がある。

### ターミナル / コマンドプロンプト

> **一言で言うと：** キーボードでコマンドを打ってPCを操作する画面

`npm install` や `mvn spring-boot:run` などのコマンドを打つ場所。  
マウスを使わずにコードで操作する。

### ビルド（Build）

> **一言で言うと：** ソースコードを実行できる形に変換する作業

TypeScriptは直接ブラウザで動かないため、JavaScriptに変換（コンパイル）する必要がある。  
`npm run build` でこの変換が行われる。

### Swagger UI（スワッガーUI）

> **一言で言うと：** APIの仕様をブラウザで確認・テストできるツール

`http://localhost:8080/swagger-ui.html` にアクセスすると、  
どんなAPIがあるか一覧で見られ、ブラウザ上から直接APIを叩いて試せる。

### Postman（ポストマン）

> **一言で言うと：** APIのリクエストをGUI操作で送れるツール

フロントエンドを作る前でも、バックエンドのAPIが正しく動いているか確認できる。  
URLとリクエストボディを入力してボタンを押すだけでレスポンスが見られる。

## 7. Gitの用語

### Git（ギット）

> **一言で言うと：** コードの変更履歴を管理するツール

「いつ・誰が・何を変えたか」を記録しておける。  
「昨日のコードに戻したい」というときに役立つ。

### GitHub（ギットハブ）

> **一言で言うと：** Gitの履歴をインターネット上に保存・共有するサービス

Gitのリモートリポジトリを管理するウェブサービス。  
チームで同じコードを共有したり、バックアップとして使う。

### リポジトリ（Repository / repo）

> **一言で言うと：** Gitで管理されているプロジェクトのフォルダ

このプロジェクトで言えば `d:/Cursor/TaskManagement/` がローカルリポジトリ。  
GitHub上にもリモートリポジトリがあり、同期させながら使う。

### コミット（Commit）

> **一言で言うと：** 変更内容を履歴として記録すること

「ファイルを保存する」のではなく「この変更を履歴に刻む」イメージ。  
コミットには必ずメッセージをつけて、何を変えたかわかるようにする。

```
git add .              ← 変更ファイルをステージング（提出準備）
git commit -m "ボードCRUDを実装"  ← 履歴として記録
```

### プッシュ（Push）

> **一言で言うと：** ローカルのコミットをGitHubに送ること

自分のPCにコミットした内容をGitHubにアップロードする。

```
git push origin develop  ← developブランチにプッシュ
```

### プル（Pull）

> **一言で言うと：** GitHubの変更を自分のPCに取り込むこと

チームで作業するとき、他の人がGitHubに上げた変更を取り込む。

```
git pull origin develop  ← GitHubのdevelopブランチの変更を取り込む
```

### ブランチ（Branch）

> **一言で言うと：** コードの「作業用コピー」

メインのコード（main）を直接触らず、  
「作業用の枝（ブランチ）」を作ってそこで開発する。  
失敗しても main が壊れない。

```
main（本番用・完成品）
 └── develop（開発用・作業はここ）
       └── feature/login（ログイン機能専用ブランチ）
```

### マージ（Merge）

> **一言で言うと：** あるブランチの変更を別のブランチに取り込むこと

`develop` で開発した内容を `main` に反映させるときに使う。

### プルリクエスト（Pull Request / PR）

> **一言で言うと：** 「このブランチをマージしてもいいですか？」という申請

チームで開発するとき、自分の変更を他の人にレビューしてもらってからマージする仕組み。

## 8. バックエンドのコード構造の用語

Spring Bootのバックエンドは、役割ごとにファイルを分けて作る。  
この構造を「レイヤードアーキテクチャ」と呼ぶ。

```
リクエスト
   ↓
Controller（コントローラー）   ← リクエストを受け取る
   ↓
Service（サービス）            ← ビジネスロジックを処理する
   ↓
Repository（リポジトリ）       ← データベースにアクセスする
   ↓
Database（データベース）       ← データを保存・取得する
```

### Entity（エンティティ）

> **一言で言うと：** DBのテーブルに対応するJavaクラス

```java
@Entity
@Table(name = "boards")
public class Board {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;
}
// このクラスが boards テーブルの1行（レコード）に対応する
```

### Repository（リポジトリ）

> **一言で言うと：** データベースへのアクセス処理をまとめた層

SQLを書かなくてもJavaのメソッドでDB操作ができる。  
Spring Data JPAが自動で実装してくれるため、インターフェース定義だけでOK。

```java
public interface BoardRepository extends JpaRepository<Board, Long> {
    // findAll(), findById(), save(), delete() などが自動で使えるようになる
}
```

### Service（サービス）

> **一言で言うと：** ビジネスロジック（業務の処理）をまとめた層

「ボードを削除するとき、紐付いているリストとカードも全部チェックする」  
のような複雑な処理はここに書く。

```java
@Service
public class BoardService {
    public Board createBoard(String title) {
        Board board = new Board();
        board.setTitle(title);
        return boardRepository.save(board);  // DBに保存
    }
}
```

### Controller（コントローラー）

> **一言で言うと：** URLへのリクエストを受け取って結果を返す層

「`GET /api/boards` にリクエストが来たら、ボード一覧を返す」という処理を定義する。

```java
@RestController
@RequestMapping("/api/boards")
public class BoardController {

    @GetMapping
    public List<BoardResponse> getBoards() {
        return boardService.findAll();
    }

    @PostMapping
    @ResponseStatus(HttpStatus.CREATED)
    public BoardResponse createBoard(@RequestBody BoardRequest request) {
        return boardService.create(request.getTitle());
    }
}
```

### DTO（ディーティーオー）

> **一言で言うと：** データの「運搬専用クラス」

Data Transfer Object の略。  
APIで受け取るデータ（Request DTO）や返すデータ（Response DTO）の形を定義する。  
Entityをそのまま外部に返すと、不要な情報やセキュリティ上の問題が出るため、DTOを使う。

```java
// リクエストDTO（フロントから送られてくるデータの形）
public class BoardRequest {
    @NotBlank(message = "タイトルは必須です")
    private String title;
}

// レスポンスDTO（フロントに返すデータの形）
public class BoardResponse {
    private Long id;
    private String title;
    private String background;
}
```

### 例外ハンドリング（Exception Handling）

> **一言で言うと：** エラーが起きたときの対処方法を定義すること

「存在しないIDのボードを取得しようとした」など、  
想定外の状況が起きたときに適切なエラーメッセージをフロントに返す仕組み。

```java
// 404エラーを返す例
throw new ResourceNotFoundException("Board not found with id: " + id);
// → フロントには { "status": 404, "message": "Board not found with id: 5" } が返る
```

## まとめ：全体の流れを再確認

```
ユーザーがブラウザで操作
         ↓
React（フロントエンド）が画面を表示・操作を受け付ける
         ↓ Axios が HTTP リクエストを送信（JSON形式）
Spring Boot（バックエンド）の Controller がリクエストを受け取る
         ↓
Service がビジネスロジックを処理
         ↓
Repository が SQL を実行（Spring Data JPA が自動生成）
         ↓
MySQL（データベース）にデータが保存・取得される
         ↓ JSON形式でレスポンスを返す
React がレスポンスを受け取り、画面を更新する
```
